# Phase 4: Streamable HTTP transport - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Add the MCP spec 2025-03-26 **Streamable HTTP** transport to FlaUI-MCP — a single
`/mcp` endpoint accepting POST (JSON-RPC) and GET (optional SSE upgrade) — usable
by modern MCP clients such as Claude Code with `"type": "http"`. The legacy
SSE transport (`/sse` + `/messages`) remains available via `--transport sse`.
Stdio transport is unchanged. All existing tools (Launch, Snapshot, Click, Type,
Fill, GetText, Screenshot, Batch) must work identically across all transports.

**Dependency note:** Roadmap declares Phase 4 depends on Phase 3
(Task Scheduler Startup), which has not yet been planned. Plan/execute for
Phase 4 should be sequenced after Phase 3 unless the user explicitly chooses
to leapfrog.

</domain>

<decisions>
## Implementation Decisions

### SDK vs hand-roll
- **D-01:** Use the official `ModelContextProtocol` SDK's built-in Streamable
  HTTP support. Wire `/mcp` via the SDK's `MapMcp()` (or current equivalent)
  rather than hand-rolling the endpoint. Researcher must verify the exact SDK
  API surface and the minimum package version that ships Streamable HTTP
  support, and confirm the tool-registration code already in the project
  works against that API.

### CLI flag value & default transport
- **D-02:** New transport value: `--transport http` selects Streamable HTTP.
- **D-02:** Default transport flips from `sse` to `http`. Existing
  `--transport sse` and `--transport stdio` remain available unchanged.
- **D-02:** Help text and any printed banners must reflect the new default.

### Co-existence model
- **D-03:** Transports are **mutually exclusive per process** — `--transport http`
  mounts only `/mcp`; `--transport sse` mounts only `/sse` + `/messages`. No
  co-mounting, no `--legacy-sse` flag. Existing SSE deployments stay on
  `--transport sse` until they explicitly migrate.

### Session-Id & stream resumability
- **D-04:** Use SDK default `Mcp-Session-Id` handling (auto-generated by the
  SDK on initialize). Do NOT bind the MCP session-id to FlaUI's `SessionManager`
  (which tracks UI-Automation windows/refs) — those remain independent concepts.
- **D-05:** Stream resumability via `Last-Event-ID` replay is **out of scope**
  for Phase 4. Accept whatever default the SDK provides (likely no replay
  buffer). See Deferred Ideas.

### Bind address & Origin protection
- **D-06:** Default Kestrel bind is `127.0.0.1` only (loopback). The server
  must reject requests whose `Origin` header is not `null`, `localhost`, or
  `127.0.0.1` (per MCP spec DNS-rebinding guidance).
- **D-06:** Add a `--bind <addr>` CLI escape hatch (e.g. `--bind 0.0.0.0`) for
  users who genuinely want LAN access. When `--bind` widens beyond loopback,
  Origin validation still applies — operator opts in to LAN routing, not to
  bypassing Origin checks.
- **D-06:** Apply the same bind + Origin policy to the legacy `--transport sse`
  path so behavior is uniform across both HTTP-based transports.

### Claude's Discretion
- Exact SDK package version and API call shape (researcher confirms).
- Internal refactor of `Program.cs` to host the new transport branch alongside
  existing SSE/stdio branches.
- Whether `Origin` validation lives as ASP.NET Core middleware vs an SDK hook.
- Exact wording of Origin-rejection responses (status code is dictated by spec,
  body text is free).
- Whether `--bind` accepts only an address or also an explicit port override
  vs reusing existing `--port`.
- Logging volume and level for new HTTP request lifecycle events.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **ASP.NET Core Kestrel host** (already wired in `SseTransport.cs`) — same
  `WebApplication` pattern can host the SDK's Streamable HTTP endpoint.
- **NLog ASP.NET Core integration** (Phase 1) — automatically applies to a
  new HTTP-mode host with no extra work.
- **`McpServer` tool dispatcher** — transport-agnostic; the SDK's Streamable
  HTTP handler will call into the same registered tools.
- **Unified CLI parser** (Phase 2) — already accommodates a transport-switch
  arg; adding `http` as a third value and a new `--bind` arg is incremental.
- **`SessionManager` / `ElementRegistry`** — per-process state, untouched by
  this phase (D-04).

### Established Patterns
- Transport selection happens in `Program.cs` after CLI parsing and before
  logging configuration finalizes (HTTP modes enable console NLog target).
- Programmatic NLog config only — no XML.
- `Console.Error.WriteLine` is forbidden in new code; use NLog logger.
- Service-mode startup sequence (Phase 2): CleanOldLogfiles → ConfigureLogging
  → Firewall → StopRunning → Install/Uninstall → Run.

### Integration Points
- `Program.cs` transport switch — add `http` branch.
- `Program.cs` argument parsing — add `--bind` value arg.
- `SseTransport.cs` (or shared helper) — apply `127.0.0.1` default bind +
  Origin validation; refactor to be reusable from the HTTP branch.
- `FlaUI.Mcp.csproj` — package version bump if needed.
- Help/usage text — update default and document `http` + `--bind`.

</code_context>

<specifics>
## Specific Ideas

- "Default flips to `http` because that's what modern MCP clients expect" —
  user explicitly chose to make the new transport the default (D-02), accepting
  that this is a soft-breaking change for existing `sse`-pinned configs.
- `127.0.0.1` + Origin check matches the MCP spec's own anti-DNS-rebinding
  guidance — user explicitly aligned with the spec rather than keeping the
  pre-existing Kestrel default.

</specifics>

<deferred>
## Deferred Ideas

- **Stream resumability** (`Last-Event-ID` replay buffer) — D-05 deferred. If
  a future client demands it, add as its own phase.
- **`SessionManager` ↔ MCP session-id binding** — D-04 alternative; would tie
  HTTP session lifecycle to FlaUI window-ref cleanup. Useful if leaked window
  handles become a problem in long-lived clients.
- **Co-mounted transports** (running `/mcp` and `/sse` on the same process) —
  rejected in D-03; could be reconsidered if a real migration scenario forces
  it.
- **Authenticated remote access** — out of scope per PROJECT.md "local trust
  boundary"; would require its own milestone if ever revisited.

</deferred>
